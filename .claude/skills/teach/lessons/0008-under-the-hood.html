Lesson 8 · Agent 的决策循环：它怎么"思考"的？

上一篇：Lesson 7 — 配置你的第一个 Hook

上节课你配置了 PostToolUse hook，能看到 agent 调用了哪些工具。但为什么它选择调用 A 而不是 B？什么时候它决定"做完了"？中间几轮发生了什么？

这节课拆开 agent 的"大脑"——决策循环（Decision Loop）。

1. Agent 不是"一次性回答"的

聊天机器人收到一句话 → 输出一句话。Agent 不一样。Agent 收到一个目标后的典型行为是：

用户："帮我找出 src/ 里所有未使用的函数"

Agent（内部循环）：
  第 1 轮 → 想："我需要先知道 src/ 下有哪些文件"          → 调 Glob
  第 2 轮 → 想："现在我需要搜索函数定义"                   → 调 Grep
  第 3 轮 → 想："找到了 47 个函数，需要检查谁调用了它们"    → 调 Grep（多次）
  第 4 轮 → 想："交叉比对后，3 个函数确实没人调用"          → 输出结果

一个任务可能涉及 N 轮的"观察 → 思考 → 行动"循环。每一轮 agent 都在决定：接下来做什么？

2. 决策循环的三个阶段

每一轮决策都经过三个阶段：

阶段agent 在做什么你能看到什么
① 观察（Observe） 接收新信息——上一步工具的输出、用户的消息、系统提示 无（内部过程）
② 思考（Think） 根据当前所有信息，推理下一步的最佳行动。要不要调工具？调哪个？还是该回复用户了？ 无（内部过程）
③ 行动（Act） 输出决定——要么调用一个工具（进入下一轮循环），要么给用户回复（循环结束） PostToolUse hook 能抓到
🔑 关键洞察

你上节课配置的 PostToolUse hook 只捕获了阶段③的一小部分（工具调用的结果面）。agent 的"思考过程"是你看不到的——除非你要求它说出来。

3. 用你已有的 hook 观察决策循环

你现在有一个现成的观察工具——上节课配置的 PostToolUse hook。来看一个真实案例：

实验：对你当前的 AI 老师说一句话，让它做一个需要多步操作的任务。比如：

Step 1 · 发起一个多步任务

回到 Claude Code 对话，输入：

帮我检查 .agents/skills/teach/ 目录下有哪些 .html 文件，统计它们的行数，告诉我哪个文件最长。

Step 2 · 查看 hook 日志

Step 3 · 数一数

你应该能看到类似这样的"行动轨迹"：

[2026-06-22 15:01:02] Called: Glob
[2026-06-22 15:01:03] Called: Read
[2026-06-22 15:01:04] Called: Read
[2026-06-22 15:01:05] Called: Read

这就是 agent 决策循环的"脚印"。Glob（列出文件）→ Read（读第一个）→ Read（读第二个）→ Read（读第三个）→ 回复。每一步都是"观察 → 思考 → 行动"的一次循环。

4. 什么决定了 agent 的决策？

Agent 的每一步决策由四样东西共同塑造：

输入作用谁设置的
System Prompt（系统提示） 定义 agent 的身份、能力边界、行为规范、可用工具列表。相当于"入职手册"。 Claude Code 框架自动注入
Skill 指令 叠加在 system prompt 之上，告诉 agent"面对这类任务时遵循这套流程"。 你（通过 SKILL.md 文件）
对话历史 用户说了什么、之前做了哪些工具调用、得到了什么结果——全部塞在上下文窗口里。 每次对话自动累积
工具输出 上一步调用的工具返回了什么信息——agent 据此决定下一步。 工具自动返回
💡 这就是为什么 hook 存在

Hook 让你在"工具调用"这个节点上插入自己的逻辑——你要么在调用之前拦截（PreToolUse），要么在调用之后观察/校验（PostToolUse）。本质上，你在修改决策循环的信息流。

5. 什么时候循环停止？

Agent 的循环不会永远跑下去。它会停止当：

  • 任务完成——agent 判断已经收集了足够信息，给用户回复
  • 上下文窗口快满——信息太多，agent 必须在忘记之前总结
  • 达到工具调用上限——Claude Code 限制单次任务最多调用一定次数工具
  • 遇到无法恢复的错误——工具反复失败，agent 放弃并报告

6. 对比：聊天机器人 vs Agent

聊天机器人

用户输入 → 模型思考 → 文本输出

1 轮，无工具，无行动

类比：笔友——收到信，回信。

Agent

用户输入 → 思考 → 调工具 → 收结果 → 思考 → 调工具 → … → 文本输出

N 轮，多工具，有行动

类比：侦探——勘查现场、采集指纹、比对记录、写报告。

7. 真实示例：你的 review skill

你项目中有一个现成的例子：.agents/skills/review/SKILL.md。这个 skill 告诉 agent 怎么做代码审查。看看它的流程：

  1. 确认基线——用 git rev-parse 和 git diff 获取变更范围
  2. 找到 spec——搜索 commit message 引用的 issue / 本地 spec 文件
  3. 并行派发子 agent——同时启动 Standards 和 Spec 两个子 agent
  4. 聚合结果——收齐两个报告，合并输出

这就是一个 多步决策循环的真实应用：每一步的输出决定下一步做什么，而且中间还分叉出并行子循环。
