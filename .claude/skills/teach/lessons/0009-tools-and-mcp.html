Lesson 9 · Tool 与 MCP：Agent 的"手"如何接入世界

上一篇：Lesson 8 — Agent 的决策循环

前面三节课你一直在学习 agent 的"大脑"——它怎么想。这节课看 agent 的"手"：它怎么行动。每个你在 hook 日志里看到的工具调用——Read、Write、Bash、Agent——都属于两种来源之一：内置 Tool 或 MCP Tool。

1. 什么是 Tool？

Tool（工具）是 agent 可以调用的具体能力。没有工具，agent 只是一张会说话的嘴——它能回答你，但不能操作文件、不能搜代码、不能执行命令。

每个工具有三个要素：

要素说明例子（Read）
名称agent 知道"这个东西叫什么"Read
输入格式调用时需要传什么参数file_path（文件路径）、offset（从第几行开始）
输出格式工具返回什么文件内容（带行号）

你 hook 日志里出现过的所有工具，都在这一张表里：

Read
读取文件内容（文本/图片/PDF）
Write
创建或覆盖文件
Edit
精确替换文件中的一段文字
Bash
执行 Shell 命令
Glob
按通配符模式匹配文件路径
Grep
在文件内容中搜索正则表达式
Agent
启动子 agent 执行任务
WebFetch
抓取网页并提取内容
WebSearch
搜索网页
TaskCreate / TaskUpdate
创建和更新任务列表

这是内置工具——Claude Code 自带的。但你还可以接入外部工具——这就引入了 MCP。

2. 为什么需要 MCP？

假设你有一个公司内部的数据库，你想让 agent 直接查它。没有 MCP 的话，你只能：

  1. 自己写一个脚本查数据库
  2. 把结果粘贴到对话里
  3. 告诉 agent "根据这个结果做分析"

有了 MCP：agent 在决策循环中可以直接调用"查数据库"这个工具，就像调用 Read 一样自然。你在 hook 日志里看到的会是：

[16:30:01] Called: Read        ← 内置工具
[16:30:03] Called: query_db    ← MCP 工具（你自己接的）
[16:30:05] Called: Write       ← 内置工具

3. MCP 的架构：USB 类比

MCP（Model Context Protocol）是一个开放协议——不是 Anthropic 专有的。它的设计理念和 USB 一模一样：

USBMCP
任何 USB 设备插入任何电脑，系统都能识别任何 MCP Server 接入任何 Agent，agent 都能调用它的工具
设备制造商只需遵循 USB 标准，不需考虑电脑品牌工具开发者只需遵循 MCP 标准，不需考虑 AI 模型品牌
U 盘、键盘、鼠标——都是 USB 设备数据库、API、文件系统——都可以是 MCP Server

架构很简单：


┌──────────┐      MCP 协议       ┌──────────────┐
│  Agent   │ ◄────────────────► │  MCP Server   │
│(Claude)  │   请求: "执行工具X"  │ (你写的/别人写的) │
│          │   响应: "结果如下"    │               │
└──────────┘                     └───────┬───────┘
                                         │
                                    ┌────┴────┐
                                    │ 数据库    │
                                    │ API      │
                                    │ 文件系统  │
                                    └─────────┘

Agent 不知道 MCP Server 内部怎么实现的——它只知道"这个 Server 提供了哪些工具、每个工具接受什么参数"。就像你的电脑不关心 U 盘里面是 NAND 还是 NOR 闪存——它只需要知道"这是一个存储设备"。

4. 内置 Tool vs MCP Tool

内置 ToolMCP Tool
谁提供的Claude Code 框架自带外部 MCP Server（你自己或第三方）
怎么配置零配置，直接用要在 settings.json 里注册 MCP Server
能干什么文件、搜索、命令、网络任何你能编程实现的事——数据库、Jira、Slack、天气预报……
例子Read, Write, Bash, Grep查 GitHub Issue、读 Google Sheets、调公司内部 API
在 agent 眼里一样的：都是一个可调用的工具一样的：都是一个可调用的工具
🔑 关键洞察

对 agent 的决策循环来说，内置工具和 MCP 工具没有区别。它在 Think 阶段看到的是一个统一的"工具菜单"——不管是框架自带的还是你通过 MCP 接入的，都是菜单上的一道菜。

5. 一个具体的 MCP 配置示例

在 settings.json 中添加一个 MCP Server：

{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "my-database": {
      "type": "stdio",
      "command": "node",
      "args": ["./mcp-servers/db-server.js"],
      "env": {
        "DB_HOST": "localhost"
      }
    }
  }
}

两种类型：

  • type: "http"——MCP Server 跑在远程服务器上，Agent 通过 HTTP 调它
  • type: "stdio"——MCP Server 跑在本地，Agent 启动一个进程然后通过标准输入/输出通信

配置完之后，重启 Claude Code——agent 的"工具菜单"里就多了你接入的那些工具。

测验：Tool 和 MCP 你弄懂了吗？

Q1. 以下哪个不是 agent 的内置工具？

A. Grep（正则搜索）
B. Glob（匹配文件）
C. query_database（查数据库）

Q2. MCP 的核心理念是什么？

A. 让 agent 跑得更快
B. 统一 agent 和外部工具的连接标准
C. 替换 agent 的内置工具

Q3. 对 agent 的决策循环来说，内置工具和 MCP 工具有区别吗？

A. 有——agent 优先用内置的

C. 没有——它们都在同一个工具菜单里

你的动手练习

  1. 打开你的 settings.local.json——看看有没有 "mcpServers" 字段（大概率没有——这就是你下一步要配的）
  2. 去 MCP 官方仓库逛一圈——github.com/modelcontextprotocol/servers，看看别人都写了什么 MCP Server（有 Slack、GitHub、Google Drive、PostgreSQL……）
  3. 看看你的 hook 日志——现在你应该能认出所有出现的工具名分别是什么能力了
  4. 心里过一遍——如果你想写一个 MCP Server 让 agent 访问你自己的数据库，你需要提供什么？（提示：工具名称、输入格式、输出格式）
